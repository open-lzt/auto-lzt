# Проектирование флоу из текста — гайд

Как из словесного описания автоматизации собрать рабочий `FlowSpec` для lzt-flow. Парный
документ к скиллу `flow-from-text` (тот делает это автоматически; здесь — принципы, чтобы делать
это осознанно, вручную или проверяя результат скилла).

## 1. Разложи задачу на канонический скелет

Почти любая автоматизация маркета/форума ложится в форму:

```
триггер → выборка/фильтр → действие → уведомление
```

- **Триггер** — что запускает флоу: событие (`новый лот`, `сообщение`), расписание (`каждый час`),
  или ручной запуск. Триггер на канвасе — отдельный узел, он помечает точку входа.
- **Выборка/фильтр** — получить данные и отсеять ненужное (`logic.get_my_lots`, `logic.compare`,
  `logic.condition`, `logic.switch`).
- **Действие** — то, ради чего всё (`market.bump`, `market.reprice`, `market.relist`, покупка,
  форумные операции через `pylzt.dynamic_call`).
- **Уведомление** — опциональный финал (сообщение/лог).

Один узел = одна работа. Если узел нельзя описать без «и» — это два узла.

## 2. Данные текут через ref, константы — через literal

- `{"ref": "fetch.item_ids"}` — вход берёт выход `item_ids` узла `fetch`.
- `{"literal": 500}` — жёстко заданное значение.
- `{"literal": "{{vars.threshold}}"}` — значение из **параметра флоу** (см. §3).

Путь можно углубить: `{"ref": "node.port[0].field"}` (см. `path.py`).

Секреты — через `env`, никогда через `literal`:

- `{"env": "FLOW_BOT_TOKEN"}` — вход берёт значение из переменной окружения хоста **по имени**.
  Значение читается на **каждом обращении** в рантайме, а не при компиляции: в скомпилированный
  `FlowIR`, в `flow.json` и в трейс попадает только имя, никогда сам секрет. Ротация токена не
  требует пересборки флоу.
- Разрешён только префикс из настройки `LZT_FLOW_FLOW_ENV_PREFIX` (по умолчанию `FLOW_`) — флоу не
  может назвать `LZT_FLOW_MASTER_KEY` или `AWS_SECRET_ACCESS_KEY` и получить их. Имя вне префикса
  или незаданная переменная → `RunFailed` (fail-closed), никогда пустая строка: пустой токен молча
  превращается в неаутентифицированный запрос.
- Токен коммить в реестр нельзя — публикуй флоу с `{"env": "FLOW_…"}`, а само значение задавай в
  окружении хоста.

⚠️ Чтение на каждом обращении значит, что значение может **измениться посреди рана** (ротация между
двумя узлами). Для токенов это правильно; для значения, которое флоу сравнивает само с собой, — нет
(тогда прочитай его один раз в узел и передавай через `ref`).

## 3. Настраиваемые значения → параметры, не россыпь литералов

Всё, что пользователь захочет менять — категория, порог цены, задержка, количество — выносится в
`FlowSpec.params` как `ParamSpec` и подключается в узле литералом `"{{vars.<key>}}"`. Тогда у флоу
есть **одно меню настроек** («Порог: 500», «Категория: steam»), а не значения, спрятанные по
разным блокам.

Контролы (`ParamControl`): `text · number · slider · toggle · select · account_picker ·
category_picker · delay`. Слайдеру задай `minimum/maximum/step`; `select` — `options`;
`category_picker` оперирует slug-ами категорий pylzt (`steam`, `fortnite`, …).

## 4. Ветвления и циклы

- **Условие**: `logic.condition` / `logic.compare` → разные `edges` по результату.
- **N-путей**: `logic.switch` с типизированным списком кейсов.
- **Цикл по лотам**: `logic.get_my_lots` → `logic.for_each_lot` (вход `item_ids` — JSON-строка
  списка int), тело — по ребру `"body"`, на каждой итерации доступен `item_id`.
- **Цикл по аккаунтам**: `logic.for_each_account`.
- **Ошибки**: `on_error` узла указывает на запасной узел — не глотай сбой молча.

## 5. Всегда валидируй

FlowSpec без проверки — не результат. Два гейта:

1. **compile** (`POST /flows/{id}/compile`) — статическая проверка: висячие рёбра, недостающие
   обязательные входы, неизвестные типы, циклы. Ошибка приходит типизированным конвертом с именем
   узла и причиной — чини по нему.
2. **dry-run** — прогон через настоящий интерпретатор на синтетических данных, без сети и БД.
   Ловит то, что компилятор не видит (форматы значений, домены).

## 6. Частые сценарии (заготовки)

| Хочу | Скелет |
|---|---|
| Поднять один лот | `market.bump(item_id={{vars.item_id}})` |
| Поднять все свои лоты | `get_my_lots → for_each_lot → bump(item_id=loop.item_id)` |
| Купить по фильтру | `триггер(новый лот) → compare(цена < {{vars.max}}) → buy` |
| Переоценить по порогу | `get_my_lots → for_each_lot → reprice` |

Больше готовых сценариев — в библиотеке шаблонов `seeds/templates/`.

См. также: скилл `.claude/skills/flow-from-text/SKILL.md` и его примеры `examples/*.json`.
