# Панель: три ловушки, найденные в браузере

Каждая выглядела как работающий код и не ловилась тестами.

---

## `Main` из `@open-lzt/ui` — это сетка с сайдбаром, а не «основная область»

**Что происходит:** один ребёнок внутри `Main` попадает в колонку шириной сайдбара, а колонка
контента остаётся пустой.

```css
.lzt-main { display: grid; grid-template-columns: var(--lzt-sidebar) 1fr; }
```

**Почему прячется:** имя. `Main` читается как семантическая обёртка контента, а не как layout
с двумя слотами. Сетка с одним ребёнком не ломается и не ругается — она молча кладёт его в первый
трек. С двумя детьми компонент ведёт себя ровно так, как ожидаешь, поэтому баг виден только там,
где сайдбара нет.

**Как поймать:** посмотреть вычисленную ширину, а не разметку. У панели `.panel-view` был
`w=240` при `max-width: 1240px` — расхождение между заданным и фактическим и есть подпись сетки.

**Что делать:** `Main` — только когда детей действительно два. Экрану без сайдбара нужен обычный
`<main>` со своим классом.

## SSE: заголовков недостаточно, чтобы соединение считалось открытым

**Что происходит:** буферизующий прокси придерживает ответ до первого байта **тела**. На тихом
канале этим байтом оказывается первый heartbeat — и «подключено» превращается в «подключено через
`heartbeat_s` секунд».

**Почему прячется:** со стороны сервера всё здорово. Статус `200` и `content-type:
text/event-stream` уходят немедленно, в логах запрос выглядит принятым. Ждёт только клиент, и
списать это легко на сеть.

**Ловушка внутри ловушки:** это **не** только про nginx. Dev-прокси vite буферизует так же, то есть
эффект есть и на localhost, где `proxy_buffering off` прописать негде.

**Замер:** индикатор жил в состоянии `connecting` 13 секунд при `heartbeat_s = 15`. После того как
поток стал слать кадр до первого ожидания — открывается сразу.

**Правило:** каждый SSE-поток пишет кадр **до** первого `await`. Комментарий (`: open`) для этого
достаточен: по спецификации он инертен — `EventSource` поднимает `open` и не отдаёт его как
сообщение.

## Мок рынка в `dev.py` отвечает одинаково на любой путь

**Что происходит:** узел, чей ответ декодируется в модель с обязательными полями, падает на
валидации. В интерфейсе это выглядит как упавшая задача, а не как проблема мока.

```
RunFailed: run ... failed at step lots: 31 validation errors for ListUserResponse
items      Field required [type=missing, input_value={'status': 'ok', 'message': 'done'}]
totalItems Field required [type=missing, input_value={'status': 'ok', 'message': 'done'}]
```

**Почему прячется:** остальным узлам плоского `{"status": "ok"}` хватает, поэтому мок годами
выглядит рабочим. Ломается ровно тот узел, который читает содержательный ответ — и ломается не при
старте, а на первом запуске флоу.

**Порядок маршрутов — часть правила:** `respx` берёт **первое** совпадение, поэтому специфичный
путь регистрируется **до** catch-all по хосту. Наоборот — и общий проглотит частный, не сообщив
об этом.

**Что делать при добавлении узла с богатым ответом:** завести path-specific маршрут в
`_maybe_mock_market`, а тело собирать `minimal_instance` из `tests/fixtures/mock_lzt_server.py` —
она строит валидный экземпляр по полям самой модели. Руками такие модели не набираются: в
`ListUserResponse` около тридцати обязательных полей, в `ListUserItem` — за сорок.

**Почему это важнее, чем кажется:** `dev.py` — то, чем проект запускают первым. Сценарий, недоступный
в нём, выглядит как сломанный продукт.
