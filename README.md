> **Переименование.** GitHub-репозиторий переименован из `lzt-flow` в **`auto-lzt`**. Пакет, модуль, CLI и почти вся документация по-прежнему используют имя **`flow`** / `lzt-flow` — это один и тот же проект, менять ничего не нужно.

**lzt-flow** — это self-hosted движок автоматизации: опиши флоу, нажми Deploy, закрой вкладку —
флоу продолжит работать на сервере, 24/7, без твоей машины.

**Авторинг — это текст.** Вы ставите готовый модуль из [официального
реестра](https://github.com/open-lzt/lzt-flows) и запускаете его из Telegram; сам движок
расширяется установкой Python-пакета — **паком узлов** (один новый тип узла) или **полным
[плагином](docs/plugins.md)** (узлы + API-роуты + бот-хендлеры + lifecycle), который владелец ставит
прямо из меню `/plugins` телеграм-бота. Бот полностью кнопочный: просмотр флоу, запуск с чтением
логов, установка модулей и плагинов. Визуальный канвас есть в этом репозитории, но он
**выключен по умолчанию** — см. *Отложено, не мертво* ниже.

[Архитектура](ARCHITECTURE.md) · [Плагины](docs/plugins.md) · [Модули](docs/modules.md) · [Флоу](docs/flow-design-guide.md) · [Доки для AI-агентов](docs/for_ai/) · [Issues](../../issues)

Референсный флоу, вокруг которого построен проект: **по расписанию → get-my-lots → for-each-account →
bump + reprice.** Продавец собирает его один раз на канвасе, жмёт Deploy, и флоу поднимает его лоты
по расписанию с сервера — переживая закрытую вкладку браузера и рестарт воркера, возобновляясь с
последнего закоммиченного шага, никогда не выполняя действие дважды.

## Быстрый старт

### Dev-режим без Docker (самый быстрый — SQLite + fakeredis + мок-маркет)

Ноль внешних сервисов. Весь основной цикл крутится в одном процессе.

```bash
uv sync --extra dev
uv run python dev.py                 # dev API на http://127.0.0.1:8000
pnpm --dir frontend dev              # канвас React Flow на http://localhost:5173
```

Проверить основной цикл end-to-end одной командой:

```bash
uv run python dev.py --demo          # прогоняет один bump-флоу -> печатает FINAL run -> {... 'status': 'completed'}
# или:  bash scripts/smoke.sh        # тот же гейт, exit 0/1 (это то, что гоняет CI)
```

### Продакшен-путь через Docker

```bash
scripts/install.sh                   # docker + compose, .env из .env.example, миграции, стек поднят
# API на http://localhost:8000 · канвас на http://localhost:5173 · health на /health
```

Поставьте перед ним reverse proxy (nginx/Caddy) для терминации HTTPS — никогда не оставляйте
деплой с настоящими токенами маркетплейса на голом HTTP.

## Управление lzt-flow

| Действие | Команда |
|---|---|
| Статус / здоровье | `curl http://localhost:8000/health` |
| Логи | `docker compose logs -f api worker` |
| Обновление | `scripts/update.sh` — тянет, мигрирует, рестартует |
| Бэкап / восстановление | `scripts/backup.sh` / `scripts/restore.sh` (Postgres) |
| Управление | UI канваса на `http://localhost:5173` — авторинг, деплой, мониторинг флоу |
| Удаление | `docker compose down -v` (сносит контейнеры + volumes; отдельного скрипта удаления пока нет) |

## Конфигурация

Скопируйте `.env.example` → `.env` и заполните секреты. Две **отдельные** поверхности конфига (осознанно):

- `LZT_FLOW_*` — собственные настройки lzt-flow, включая `LZT_FLOW_MASTER_KEY` (envelope-ключ для
  токенов аккаунтов).
- `LZT_*` — встроенный движок `lzt-eventus` (нужен только для пути триггера `on-event`), включая
  `LZT_TOKENS` (токены для поллинга) и `LZT_TOKEN_ENC_KEY` — ключ, **отличный** от
  `LZT_FLOW_MASTER_KEY`.

### Песочница против фейкового маркет-бэкенда

`LZT_FLOW_MARKET_BASE_URL` (по умолчанию не задан, реальный `prod-api.lzt.market`) направляет каждое
**эфемерное** создание `MarketAdapter` на другой маркет-бэкенд вместо реального — чтобы гонять
lzt-flow против фейкового API во время локальной разработки/ручного тестирования, не трогая реальный
маркетплейс. Это отличается от dry-run-гейта ниже: с этой переменной реальный код всё так же
выполняет реальные HTTP-запросы, просто против фейкового сервера.

Поднимите фейковый сервер из соседнего репозитория `lzt-testnet` (полный quickstart — в его
собственном README) и направьте на него `dev.py`:

```bash
cd ../lzt-testnet && scripts/run.sh          # запускает мок-маркет на 127.0.0.1:8765
# в этом репозитории:
LZT_FLOW_MARKET_BASE_URL=http://127.0.0.1:8765 uv run python dev.py --demo
```

**Известный пробел**: путь создания через пул-клиент (`MarketAdapter(client=...)`, используется там,
где `httpx.AsyncClient` расшарен/инжектится) **не** учитывает эту переменную окружения — только
эфемерный путь создания на каждый вызов. Это документированное ограничение, а не тихое.

## Примеры

Два независимых способа работать с уже запущенным инстансом — выбирайте по поверхности интеграции,
а не как прогрессию.

### REST API — авторинг, компиляция и запуск флоу императивно
Для скриптования разового запуска или вашей собственной автоматизации напрямую против API.

```bash
# examples/01_rest_flow.sh
curl -sS -X POST http://localhost:8000/flows/create \
  -H 'Content-Type: application/json' \
  -d '{
        "name": "bump-once",
        "entry_node_id": "bump",
        "nodes": [{"id": "bump", "type": "market.bump", "inputs": {"item_id": {"literal": 123456}}}]
      }'
# -> {"flow_id": "..."}

curl -sS -X POST "http://localhost:8000/flows/<flow_id>/compile"
# -> {"flow_ir_id": "...", "node_count": 1}

curl -sS -X POST http://localhost:8000/runs/create \
  -H 'Content-Type: application/json' -d '{"flow_id": "<flow_id>"}'
# -> {"run_id": "...", "status": "pending"}

curl -sS "http://localhost:8000/runs/<run_id>/get"
# -> {"run_id": "...", "status": "completed"}
```
`X-API-Key` обязателен на мутирующих вызовах выше, как только задан `LZT_FLOW_API_KEY` (по умолчанию
пусто, что нормально для self-host только на loopback).

#### Параметры и синхронный invoke

Флоу может объявить **поверхность параметров** — плоский набор настроек (задержка, количество,
категория), редактируемых из одной формы настроек вместо поиска значений внутри отдельных блоков.
Объявите их под `params` и ссылайтесь на каждый из входа узла как на литерал `"{{vars.<key>}}"`:

```jsonc
{
  "name": "bump-one",
  "entry_node_id": "bump",
  "params": [{ "key": "item_id", "label": "ID лота", "control": "number", "required": true, "default": 1 }],
  "nodes": [{ "id": "bump", "type": "market.bump", "inputs": { "item_id": { "literal": "{{vars.item_id}}" } } }]
}
```

И `POST /runs/create`, и `POST /flows/{id}/invoke` принимают тело `params`, проверяемое против
объявленной поверхности. `invoke` запускает флоу **синхронно** и возвращает его финальный вывод —
ограничено `LZT_FLOW_FLOW_INVOKE_TIMEOUT_S` (по умолчанию `60`); для долгих флоу используйте
асинхронный путь `/runs/create`.

```bash
curl -sS -X POST "http://localhost:8000/flows/<flow_id>/invoke" \
  -H 'Content-Type: application/json' -d '{"params": {"item_id": 123456}}'
# -> {"run_id": "...", "status": "completed", "output": {...}}
```

### Триггеры — привязать расписание, чтобы флоу срабатывал без дальнейших вызовов
Для настоящего use case автопилота: никакого императивного вызова после настройки, флоу запускает себя сам.

```bash
# examples/03_attach_schedule_trigger.sh
curl -sS -X POST "http://localhost:8000/flows/<flow_id>/triggers/create" \
  -H 'Content-Type: application/json' \
  -d '{"kind": "schedule", "schedule_cron": "*/30 * * * *"}'

curl -sS "http://localhost:8000/flows/<flow_id>/status"
# -> {"running": true, "active_accounts": N, "last_run_at": "..."}
```

## Архитектура

lzt-flow — это тонкий доменный слой над переиспользуемой экосистемой `lzt-*` — `pylzt`
(транспорт) и `lzt-eventus` (событийная фабрика) **переиспользуются, а не переписываются**. См.
**[ARCHITECTURE.md](ARCHITECTURE.md)** — полная сравнительная таблица (vs. клиентский flow-билдер)
и диаграмма рантайма.

Проектируете флоу из текстового описания? См. **[гайд по проектированию флоу](docs/flow-design-guide.md)**
и скилл `flow-from-text` (`.claude/skills/flow-from-text/`), который превращает описание на
обычном языке в проверенный компиляцией и dry-run `FlowSpec`.

> `pylzt` и `lzt-eventus` (организация `open-lzt`) — отдельные приватные репозитории,
> запиненные по SHA в `pyproject.toml` — одного клонирования этого репозитория недостаточно для
> сборки, пока у вас нет доступа на чтение.

### О доверии к модулю

`sha256` модуля в `index.json` реестра — это **целостность передачи**: он доказывает, что
устанавливаемый вами флоу байт-в-байт совпадает с тем, что был отревьюен. Это **не подпись**, и он
ничего не говорит о том, можно ли доверять его автору. Единственное, что стоит между вами и
враждебным модулем, — это то, что мейнтейнер прочитал его перед мержем, а модуль — это *данные*, так
что вы тоже можете его прочитать.

## Плагины узлов

Узел добавляется установкой дистрибутива, который объявляет entry point `lzt_flow.nodes`:

```toml
# в pyproject.toml вашего пакета
[project.entry-points."lzt_flow.nodes"]
my_pack = "my_pack.nodes:REGISTRATIONS"
```

`pip install` — это весь шаг установки. Нет папки плагинов для сканирования и пути для настройки,
поэтому узел не может появиться без того, чтобы кто-то установил пакет, который его предоставляет.
Затем узел появляется в `GET /catalog/list`, получает форму в веб-канвасе и в боте без единой правки
в обоих, и компилируется и выполняется тем же интерпретатором, что и встроенные узлы.

Два правила, которые плагин не может обойти:

- **Он никогда не может подменить встроенный узел.** Плагин, заявляющий `market.bump`, роняет старт
  с `DuplicateNodeType`, называя обе стороны. Он никогда не побеждает порядком загрузки — пакет,
  тихо подменивший денежный узел, заставил бы все флоу на стенде вызывать его код, без единого следа.
- **Он не может неконтролируемо ходить в сеть.** Узел-запрос наследуется от `BaseRequestNode`, чей
  `execute()` финальный и проходит через `deps.http`, который нельзя сконструировать без
  `EgressPolicy`. Обойти это негде.

Каждый узел объявляет, что он умеет (`NodeCapability`), и именно по этому объявлению фильтрует
валидатор модулей — см. `app/domain/catalog/capabilities.py`.

## Исходящие запросы

Узлы, которые говорят с чем-то, кроме маркетплейса, проходят через egress-забор
(`app/domain/egress/`). Только `https`, сверка с **точным** allow-list, который **пуст по
умолчанию**, оценка *разрешённого адреса*, а не хостнейма, и подключение к тому же адресу, который
был проверен — так что `2130706433`, `0177.0.0.1` и `::ffff:127.0.0.1` все распознаются как loopback,
и DNS rebinding нечего переразрешать. Редиректы отклоняются, а не следуются.

```bash
LZT_FLOW_EGRESS_ALLOWED_HOSTS=api.telegram.org   # через запятую; пусто значит "никуда не достучаться"
```

Ненастроенное значит недостижимое, осознанно: этот процесс сидит рядом с Redis, который хранит
гарды идемпотентности денег и очередь задач, поэтому `http://redis:6379` в модуле сообщества иначе
был бы выполнением кода в воркере.

## Отложено, не мертво

Визуальный канвас и UI авторинга композитных блоков — за флагом `VITE_BUILDER_ENABLED`, **выключены
по умолчанию**. Код осознанно остаётся в дереве — это отсрочка, а не удаление, и проход `/cleanup`,
который заключит, что `AuthoringMode` и путь деплоя — мёртвый код, удалил бы рабочую фичу. Соберите
с включённым флагом, чтобы вернуть их:

```bash
VITE_BUILDER_ENABLED=1 npm run build
```

Выключение UI флагом — это **продуктовое решение, а не граница безопасности**: мутирующие эндпойнты
всё ещё на месте и всё так же под ключом API. Что делает превью-сборку честной — она не поставляет
ключ, который прятался бы за кнопками: ключ вводится оператором в рантайме в sessionStorage.
